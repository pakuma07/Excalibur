# Amazon S3 (Simple Storage Service)

> Amazon Simple Storage Service — launched **March 2006**, the first public AWS service. No single seminal academic paper; design knowledge comes from AWS documentation, re:Invent talks, the *Amazon Builders' Library*, and the 2020 strong-consistency announcement. (Conceptual ancestor: the *Dynamo* paper, SOSP 2007 — see the Dynamo case study.)

---

## Overview

Amazon S3 is a **planet-scale object storage service**: a highly durable, highly available, virtually unlimited store for *objects* (blobs of bytes plus metadata), accessed over a simple HTTP/REST API. It was AWS's first product and remains the canonical example of a **cloud storage primitive** that thousands of other systems (and AWS services) are built on.

S3 is not a filesystem and not a block device. It is a **key → object** store with a **flat keyspace per bucket**, optimized for **durability and scale above all**, exposed through a small, stable, versioned HTTP API.

| | |
|---|---|
| **Built by** | Amazon Web Services |
| **Launched** | March 14, 2006 |
| **Design goal headline** | 99.999999999% (**"11 nines"**) durability |
| **Scale (publicly stated)** | Hundreds of trillions of objects; tens of millions of requests/sec |
| **Major consistency change** | December 2020: **strong read-after-write consistency** for all operations |

---

## The Problem It Solved

Before S3, anyone needing to store large amounts of data reliably had to **buy, rack, replicate, monitor, and capacity-plan storage hardware** — and getting durability right (RAID, backups, multi-site replication, bit-rot detection) is genuinely hard. S3's pitch: storage as a **utility** — pay per GB and per request, infinite capacity, someone else owns durability and availability, accessed by a trivial HTTP API from anywhere.

The hard design constraints:

- **Durability is paramount.** Losing customer data is unacceptable; the target is 11 nines (statistically, you'd expect to lose a single object once every ~10 million years for a given set of 10,000 objects).
- **Effectively unlimited scale** with no capacity planning by the customer, and **graceful** growth on the provider side.
- **High availability** independent of any single disk, server, rack, or even data center.
- **A simple, stable API** that can never break, because millions of applications depend on it indefinitely.

## Object Model & Flat Keyspace

S3's data model is intentionally minimal:

```
Account
 └── Bucket (globally unique name, lives in one Region)
       └── Object   key (string, up to 1024 UTF-8 bytes)  ->  value (bytes, up to 5 TB)
                     + metadata (system + user, e.g. Content-Type)
                     + optional version ID
                     + ETag (typically an MD5 of the data)
                     + storage class
```

- **Buckets** are top-level containers, scoped to an AWS **Region**; bucket names are globally unique.
- **Objects** are addressed by a **key**, an opaque string. Objects are **immutable as a unit** — you can't append to or patch an existing object; a `PUT` replaces the whole object (a new version if versioning is on).
- **The keyspace is FLAT.** There are no real directories. The `/` characters in a key like `2026/06/logs/app.log` are pure convention; the console *renders* "folders" by treating `/` as a delimiter in `LIST` results (`prefix` + `delimiter`), but physically the key is just a string. This flatness is what lets the keyspace partition arbitrarily across the fleet.

This flat key → blob model is *the* lesson many imitators copy: the simplest possible abstraction that still covers the use cases, leaving structure (folders, hierarchy) as a client-side rendering concern.

## Architecture

S3 is a collection of **micro/sub-services** behind a stateless front-end fleet. AWS hasn't published a full architecture paper, but the publicly described shape is:

```mermaid
flowchart TB
    CLIENT[Clients / SDKs\nHTTP REST: GET/PUT/...]
    DNS[DNS / global routing]

    subgraph Region
      LB[Front-end fleet\nstateless API/web servers\nauth, throttling, request routing]

      subgraph Control
        IDX[Indexing / Metadata subsystem\nkey -> object location\n(sharded, strongly consistent)]
      end

      subgraph DataPlane
        STORE[Storage node fleet\n(thousands of disks)\nspread across 3+ Availability Zones]
        EC[Durability engine\nerasure coding / replication\nacross AZs]
      end

      KEYMAP[Keymap / placement\n(which shards hold the object)]
    end

    CLIENT --> DNS --> LB
    LB --> IDX
    IDX --> KEYMAP
    LB --> STORE
    STORE --> EC
    EC -. spread across .-> AZ1[(AZ 1)]
    EC -. .-> AZ2[(AZ 2)]
    EC -. .-> AZ3[(AZ 3)]
```

Key structural ideas:

- **Stateless front end** does authentication (SigV4 signatures), authorization (IAM, bucket policies, ACLs), throttling, and **request routing**. Any front-end node can serve any request, so the fleet scales horizontally and tolerates node loss.
- A **metadata/indexing subsystem** maps `(bucket, key[, version])` → the physical location(s) of the object's data shards. This index must be **strongly consistent** (it's what made the 2020 consistency change possible).
- A **storage node fleet** holds the actual bytes on huge numbers of disks, **spread across at least three Availability Zones** within a Region. AZs are physically separate data centers with independent power/network, close enough for low-latency synchronous replication.

## How It Works

### Durability — the "11 nines" engine

Durability comes from **redundancy across many disks and multiple Availability Zones**, plus continuous integrity checking. S3 standard storage stores object data **redundantly across a minimum of three AZs** in a Region, so the loss of an entire data center does not lose data.

Two redundancy techniques (S3 has used both; modern large objects favor erasure coding for cost):

- **Replication:** keep N full copies on independent disks/AZs.
- **Erasure coding (Reed-Solomon):** split an object into `k` data shards plus `m` parity shards (e.g., split into shards such that any `k` of `k+m` can reconstruct the object). This gives the *same or better* durability as N-way replication at a **fraction of the storage overhead** — instead of 3× (200% overhead) you might pay ~1.4–1.8× while tolerating multiple simultaneous shard losses. Shards are distributed across AZs so that no single AZ failure prevents reconstruction.

Durability is **actively maintained, not just initially achieved:**

- **Continuous integrity verification:** S3 regularly checksums stored data, detects bit-rot / failed disks, and **re-replicates or re-encodes** to restore the redundancy level. AWS describes designing the system to **sustain the concurrent loss of data in two facilities** without data loss.
- **Checksums end-to-end:** uploads can carry checksums (MD5/CRC/SHA); S3 verifies on `PUT` and on internal transfers, so silent corruption is caught.

> Note: **Durability (11 nines) ≠ Availability.** S3 Standard targets **99.99% availability**, with an SLA; durability is the much stronger 11-nines property. They are different promises — *we won't lose your data* vs. *you can reach it right now*.

### Partitioning & request routing

- The **flat keyspace of each bucket is partitioned** across many backend partitions. S3 dynamically **splits hot partitions** as request rate or key density grows.
- Historically, S3 partitioned by **key prefix**, which is why old guidance told you to add high-entropy/random prefixes to keys to spread load and avoid a hot partition (e.g., monotonically increasing timestamp keys all landing in one partition). In **2018 S3 improved automatic partitioning** so that each *prefix* now sustains very high request rates (e.g., **3,500 PUT/POST/DELETE and 5,500 GET/HEAD requests per second per prefix**), and adding prefixes multiplies throughput — largely removing the old "randomize your key prefix" advice. The deep lesson — *your key design determines how load distributes* — is identical to Bigtable/Dynamo hot-spotting.

### Consistency — from eventual to strong (2020)

This is one of S3's most important and instructive evolutions.

| Era | Consistency model |
|---|---|
| 2006 – Nov 2020 | **Read-after-write consistency for new-object PUTs**, but **eventual consistency** for overwrite `PUT`s, `DELETE`s, and (notoriously) `LIST`. After an overwrite or delete, a subsequent `GET`/`LIST` might briefly return the *old* version or a stale listing. There was also a famous caveat: a `GET` issued *before* an object existed (a negative cache) could remain eventually-consistent even for new objects. |
| **Dec 1, 2020 onward** | **Strong read-after-write consistency for ALL operations** — `GET`, `PUT`, `LIST`, overwrites, and deletes — automatically, at no extra cost or performance penalty. After a successful write, any subsequent read is guaranteed to see the latest data, and `LIST` immediately reflects changes. |

Why it mattered: the original eventual model was a constant source of bugs and forced ecosystems to build workarounds — most famously **S3Guard** (a DynamoDB-backed consistent index) for Hadoop/Spark and EMRFS consistent view for EMR, because data-processing jobs that wrote files and then immediately listed/read them could miss data. The 2020 change to strong consistency **eliminated the need for these workarounds**.

Achieving strong consistency at S3's scale (it was retrofitted into a 14-year-old eventually-consistent system without changing the API, the price, or the performance) is a landmark engineering achievement. It rests on a **strongly consistent metadata/index subsystem** that, on a read, authoritatively resolves the current location/version of an object before fetching bytes — turning what was once a best-effort cache into an authoritative, replicated index.

### Multipart upload

For large objects, a single `PUT` is fragile (one network blip restarts the whole transfer) and the per-request size is capped (5 GB for a single `PUT`; objects up to **5 TB** require multipart). **Multipart upload** breaks an object into parts:

```
1. CreateMultipartUpload(bucket, key)          -> uploadId
2. UploadPart(uploadId, partNumber=1, bytes)   -> ETag   ┐ parts can be uploaded
   UploadPart(uploadId, partNumber=2, bytes)   -> ETag   �develop in PARALLEL, retried
   ...                                                    ┘ independently
3. CompleteMultipartUpload(uploadId, [ {partNumber, ETag}, ... ])
   -> S3 assembles the parts into one object atomically
   (or AbortMultipartUpload to discard)
```

Benefits: **parallel, resumable, retryable** uploads; better throughput; and the ability to begin uploading before the total size is known. The final object's `ETag` reflects the multipart assembly (not a simple MD5).

### Storage classes (tiering for cost/access trade-offs)

S3 exposes multiple **storage classes**, all with the same API and the same 11-nines durability, trading **access latency/availability/retrieval cost vs. storage price**:

| Class | Use case / characteristic |
|---|---|
| **S3 Standard** | Frequent access; lowest latency; multi-AZ; 99.99% availability target. |
| **S3 Intelligent-Tiering** | Auto-moves objects between access tiers based on observed access patterns; no retrieval fees; good when access is unpredictable. |
| **S3 Standard-IA** (Infrequent Access) | Cheaper storage, retrieval fee; still multi-AZ. |
| **S3 One Zone-IA** | Cheaper still, stored in a **single AZ** (lower durability vs. AZ loss) — for re-creatable data. |
| **S3 Glacier Instant Retrieval** | Archive with millisecond access. |
| **S3 Glacier Flexible Retrieval** | Archive; minutes-to-hours retrieval; very cheap. |
| **S3 Glacier Deep Archive** | Lowest cost; hours-long retrieval; long-term cold archive. |

Lifecycle policies transition objects between classes (e.g., Standard → IA after 30 days → Glacier after 90).

## Key Innovations / What Made It Special

1. **Storage as a utility primitive** — infinite, durable, pay-per-use storage behind a trivial HTTP API. It arguably *kicked off the public cloud.*
2. **Durability as the headline contract** (11 nines), actively maintained via multi-AZ redundancy + erasure coding + continuous integrity verification.
3. **The flat key → immutable-blob model**: the minimal abstraction that scales, with hierarchy left as a client-side rendering concern.
4. **API stability as a feature.** The S3 API is so stable and ubiquitous it became a **de facto industry standard** — countless systems and competitors implement "the S3 API."
5. **Retrofitting strong consistency** into a planet-scale, 14-year-old eventually-consistent system with no API/price/perf regression (2020).

## Data Model / APIs

A REST API over HTTP; objects addressed as `https://<bucket>.s3.<region>.amazonaws.com/<key>`:

```http
PUT    /my-key            # upload (create or overwrite) an object
GET    /my-key            # download an object (supports Range: bytes=…)
HEAD   /my-key            # fetch metadata only
DELETE /my-key            # delete (or add a delete marker if versioned)
GET    /?list-type=2&prefix=2026/06/&delimiter=/   # LIST objects (paginated)

# Multipart
POST   /my-key?uploads                 # CreateMultipartUpload
PUT    /my-key?partNumber=N&uploadId=… # UploadPart
POST   /my-key?uploadId=…              # CompleteMultipartUpload
```

Surrounding features: **versioning**, **lifecycle policies**, **server-side encryption** (SSE-S3/SSE-KMS), **bucket policies + IAM + ACLs + Block Public Access**, **presigned URLs**, **Cross-Region/Same-Region Replication**, **event notifications** (→ Lambda/SQS/SNS), **S3 Select** (query inside objects), **Object Lock** (WORM/compliance), and request authentication via **SigV4** signatures.

## Trade-offs & Limitations

| Trade-off | Consequence |
|---|---|
| Object store, **not a filesystem** | No partial in-place updates/appends, no rename (rename = copy + delete), no POSIX semantics. Object is the unit of write. |
| **Per-object overwrite only** | To "edit" you re-PUT the whole object (a new version). |
| **LIST is paginated & can be costly** | Listing a bucket with billions of keys is slow/expensive; design keys/prefixes for the access pattern, not for browsing. |
| **Latency** | Higher and more variable than local disk/EBS; throughput-oriented. Not for low-latency random IO of a block device. |
| **Eventual-consistency era foot-guns (pre-2020)** | Caused real data-loss-looking bugs; resolved, but a reminder that weak consistency leaks into application correctness. |
| **Hot-key / hot-prefix** | Improved by auto-partitioning, but extreme skew still needs key design awareness. |
| **Cost model has many dimensions** | Storage + requests + data transfer + retrieval (for cold classes) + management features — easy to mis-estimate. |

## Influence & Legacy

- **Defined cloud object storage** as a category. Azure Blob Storage, Google Cloud Storage, and others followed the same key→blob, multi-class, HTTP-API model.
- **The S3 API is a de facto standard.** MinIO, Ceph (RADOS Gateway), Backblaze B2, Cloudflare R2, Wasabi, and many on-prem systems implement S3-compatible APIs so existing tooling "just works."
- **Foundation of the modern data lake / lakehouse.** Parquet/ORC on S3 + query engines (Athena/Presto/Trino, Spark, Snowflake, Databricks, Redshift Spectrum) and table formats (**Apache Iceberg, Delta Lake, Apache Hudi**) all assume an S3-like object store as the durable substrate.
- It is the **storage backbone of AWS itself** — countless AWS services persist to S3.
- Conceptually it shares DNA with the **Dynamo** paper (highly available, eventually-consistent, internet-scale), and its 2020 strong-consistency move mirrors the broader industry arc back toward strong guarantees once they became affordable at scale (cf. Spanner).

## Lessons for Architects

1. **Pick the smallest abstraction that covers the use cases.** Flat key → immutable blob is almost embarrassingly simple, and that simplicity is exactly why it partitions, scales, and endures. Resist baking in hierarchy/structure the substrate doesn't need.
2. **Durability and availability are different promises — state each explicitly.** S3 advertises 11 nines durability *and* 99.99% availability; conflating them misleads users. Decide and communicate each independently, and actively *maintain* durability (continuous integrity checks + re-replication), don't just achieve it at write time.
3. **Erasure coding beats replication for cost at scale** when you can tolerate the encode/decode cost and the slightly higher reconstruction complexity — same durability, far less storage overhead. Spread shards across independent failure domains (AZs).
4. **Weak consistency leaks into application correctness.** S3's eventual-consistency era spawned an entire cottage industry of workarounds (S3Guard). If you can afford strong consistency, it removes a whole class of customer bugs — and customers will quietly build fragile workarounds if you don't.
5. **API stability is a feature you ship forever.** A storage primitive's API is a multi-decade contract; design it to be small, versioned, and stable. Stability is what let S3 become an industry standard *and* what let AWS re-engineer the internals (consistency, partitioning, erasure coding) repeatedly **without breaking a single client.**
6. **Make the hot path stateless and route on top of an authoritative index.** A stateless front-end fleet + a strongly-consistent metadata/index layer is what allowed both horizontal scale *and* the later retrofit of strong consistency.
7. **Key/prefix design determines load distribution** — the same hot-spotting lesson as Bigtable, Dynamo, and Cassandra. The naming scheme *is* part of the performance architecture.
